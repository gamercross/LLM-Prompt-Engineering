## 10. 비용 제약 환경에서의 운영 전략

무료 LLM을 사용할 경우에는 모델 성능보다 프롬프트 품질이 더욱 중요하다. 동일한 질문을 여러 번 반복하기보다 한 번의 요청으로 필요한 정보를 최대한 얻을 수 있도록 프롬프트를 설계하였다.

### 개선 전략

- 입력 템플릿을 고정하여 매번 동일한 형식으로 요청한다.
- Few-shot 예시를 제공하여 출력 형식을 일관되게 유지한다.
- 단계적 접근(요구사항 확인 → 부족한 정보 질문 → 결과 생성)을 적용하여 재요청 횟수를 줄인다.
- 긴 문서는 필요한 부분만 입력하여 토큰 사용량을 최소화한다.
- 동일한 업무는 프롬프트 템플릿을 재사용하여 작업 시간을 단축한다.

### 기대 효과

- 무료 모델에서도 출력 품질 향상
- 불필요한 재생성 감소
- 토큰 사용량 절감
- 업무 자동화 효율 증가

## 11. 업무 정책 변경 시 환각 방지 전략

업무 규정이나 정책은 자주 변경될 수 있으므로 LLM의 학습 데이터만 신뢰하지 않는다.

### 운영 원칙

1. 최신 정책 문서를 항상 입력 데이터로 제공한다.
2. 정책 문서가 없는 경우에는 확답하지 않는다.
3. "확인이 필요합니다."를 우선 출력하도록 시스템 프롬프트를 설계한다.
4. 정책 번호나 시행일이 있으면 함께 표시한다.
5. 최신 정책 여부를 확인하도록 사용자에게 안내한다.

### 예시

사용자
> 출장비 규정이 어떻게 되나요?

좋은 답변

> 최신 규정 문서가 제공되지 않아 정확한 답변이 어렵습니다.
> 최신 출장비 규정 또는 사내 위키를 확인한 후 요약해 드릴 수 있습니다.

이와 같은 운영 전략을 통해 사실 오류와 환각을 최소화할 수 있다.

## 12. 10턴 이상 문맥 유지를 위한 프롬프트 개선 원리

10턴 이상의 대화에서는 이전 요구사항을 지속적으로 기억하는 것이 중요하다.

본 프로젝트에서는 다음과 같은 구조를 적용하였다.

### 1. 요구사항 고정

대화 초기에 다음 내용을 먼저 확정한다.

- 출력 형식
- 말투
- 대상
- 금지어
- 필수 포함 항목

이후에는 변경 요청이 없는 한 계속 유지한다.

---

### 2. 변경 사항만 수정

사용자가

> 말투를 조금 더 부드럽게 해줘.

라고 요청하면

말투만 변경하고

회의 내용

출력 형식

결정사항

Action Items

등은 그대로 유지하도록 설계하였다.

---

### 3. 확인 질문 우선

정보가 부족하면 바로 답하지 않고

최대 3개의 확인 질문을 먼저 수행하도록 하였다.

이를 통해 잘못된 추측을 줄였다.

---

### 4. 단계적 처리

최종 답변 전에 내부적으로

① 요구사항 확인

② 누락 정보 확인

③ 요약 생성

④ 형식 검토

순서로 처리하도록 프롬프트를 설계하였다.

최종 출력에서는 내부 추론을 노출하지 않고 핵심 결과만 제공하였다.

---

### 개선 효과

- 긴 대화에서도 문맥 유지
- 조건 변경 시 필요한 부분만 수정
- 환각 감소
- 출력 형식 일관성 향상

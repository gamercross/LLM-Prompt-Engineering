# 🤖 Chapter-1 프로젝트 소개

# 📖 Project Overview

## 프로젝트 소개

최근 생성형 AI는 다양한 업무에서 활용되고 있지만,
같은 질문을 입력하더라도 사용하는 모델과 프롬프트 설계 방식에 따라
결과물의 품질은 크게 달라진다.

특히 업무 자동화에서는 단순히 답변을 생성하는 것이 아니라,

- 정확성
- 일관성
- 문맥 유지
- 출력 형식 준수
- 환각(Hallucination) 최소화

가 매우 중요한 요소가 된다.

본 프로젝트는 이러한 문제를 해결하기 위해 **Prompt Engineering 기법**을 적용하여
회의록 요약 업무를 자동화하는 것을 목표로 한다.

또한 동일한 업무를 여러 LLM에게 수행하도록 하여
각 모델의 장단점을 비교하고,
가장 적합한 모델을 선정하는 과정을 수행하였다.

---

# 🎯 Project Goal

이번 프로젝트의 목표는 다음과 같다.

### 1️⃣ 여러 LLM 모델 비교

동일한 업무를

- ChatGPT
- Claude
- Gemini

에게 수행시켜 결과를 비교한다.

---

### 2️⃣ Prompt Engineering 적용

단순 질문이 아닌

- Persona
- System Prompt
- User Prompt
- Few-shot
- 단계적 추론(Reasoning Prompt)

기법을 적용하여 출력 품질을 향상시킨다.

---

### 3️⃣ Hallucination 최소화

사실 기반 질문에서

- 틀린 정보를 단정하지 않도록 하고
- 근거를 제시하거나
- 모르면 모른다고 답하도록 설계한다.

---

### 4️⃣ 업무 자동화

회의록을 입력하면

- 결정사항
- Action Items
- 리스크
- 후속 일정

을 자동으로 정리할 수 있도록 한다.

---

### 5️⃣ 문맥 유지

10턴 이상의 대화에서도

- 기존 요구사항을 유지하고
- 변경사항만 반영하는 구조를 설계한다.

---

# 🚨 Problem Definition

## 기존 업무의 문제점

회의가 끝난 이후 작성되는 회의록은

- 내용이 길고
- 핵심 내용이 분산되어 있으며
- 담당자와 일정이 명확하지 않은 경우가 많다.

또한 프로젝트가 여러 개 동시에 진행될 경우

회의록을 읽고

- 결정사항
- 해야 할 일
- 일정
- 위험 요소

를 직접 정리하는 데 많은 시간이 소요된다.

---

### ❌ 기존 방식의 문제

|문제|설명|
|:---|:---|
|⏳ 시간 소모|회의록을 직접 읽고 정리해야 한다.|
|📄 형식 불일치|작성자마다 정리 방식이 다르다.|
|❗ 중요 내용 누락|Action Item이 빠질 수 있다.|
|👥 담당자 혼동|누가 무엇을 하는지 확인이 어렵다.|
|🔍 검색 어려움|핵심 내용만 빠르게 찾기 어렵다.|

---

### 💡 해결 방법

LLM을 활용하여

회의록을 입력하면

자동으로

- 결정사항
- 해야 할 일
- 리스크
- 일정

을 추출하도록 Prompt를 설계하였다.

---

# 💼 Task Selection

## 선택한 업무 과업

### 📌 회의록 요약 자동화

본 프로젝트에서는

**회의록 요약 업무**

를 자동화 대상으로 선정하였다.

---

### 선택 이유

회의록은

- 대부분의 기업에서 자주 발생하며
- 반복 업무 비중이 높고
- LLM이 가장 잘 수행하는 업무 중 하나이다.

또한

정확성,

문맥 유지,

출력 형식,

환각 방지

등

이번 과제에서 요구하는 대부분의 평가 항목을
확인하기 적합하다.

---

## 👨‍💼 Target User

|항목|내용|
|:---|:---|
|사용자|프로젝트 매니저(PM)|
|사용 환경|기업 내부 회의|
|목적|회의록 자동 요약|
|주요 업무|결정사항 정리 및 업무 공유|

---

## 📥 Input Data

입력은 자유 형식의 회의 메모 또는 회의록이다.

예시

```text
프로젝트 킥오프 회의

- 디자인 일정은 다음 주까지 완료
- 개발은 5월부터 시작
- 예산은 1,000만원 승인
- QA 일정은 추후 협의
```

---

## 📤 Output Format

출력은 반드시 아래 형식을 따른다.

```text
📌 결정사항

📌 Action Items

📌 리스크

📌 후속 일정

📌 확인 필요 사항
```

출력 형식을 고정함으로써
모든 회의록을 동일한 형식으로 관리할 수 있도록 설계하였다.


# 🏆 최종 선정

## 선정 모델

# 🥇 ChatGPT (GPT-5.5)

---

## 선정 이유

### 1️⃣ 가장 안정적인 출력 형식

회의록 자동화에서는 항상 동일한 형식으로 출력하는 것이 중요하다.

ChatGPT는

- 결정사항
- Action Items
- 리스크
- 후속 일정

을 요구한 형식 그대로 유지하였다.

---

### 2️⃣ 환각(Hallucination)이 가장 적었다.

회의 내용에 없는 정보를 추가하지 않았으며,

정보가 부족한 경우

> "확인이 필요합니다."

라고 명확하게 표시하였다.

이는 업무 자동화에서 매우 중요한 장점이다.

---

### 3️⃣ 긴 문맥 유지 능력이 우수하였다.

회의록은 일반적으로 수백 줄 이상의 긴 문서가 될 수 있다.

ChatGPT는 긴 문맥에서도

- 결정사항
- 일정
- 담당자

를 정확하게 구분하였다.

---

### 4️⃣ Prompt Engineering 적용 효과가 가장 컸다.

Persona,

Few-shot,

System Prompt,

Reasoning Prompt

를 적용하였을 때 출력 품질 향상이 가장 크게 나타났다.

---

# 💡 분석 및 느낀 점

이번 비교를 통해 동일한 입력을 사용하더라도 모델에 따라 결과 품질이 크게 달라질 수 있음을 확인하였다.

특히 업무 자동화에서는 단순히 문장을 자연스럽게 생성하는 능력보다,

- 정확성
- 형식 준수
- 환각 방지
- 문맥 유지

가 더욱 중요한 평가 요소임을 확인하였다.

# ⚙️ Chapter 3. 시스템 설계 (System Design)

# 🧠 Chapter 4. Prompt Engineering & Few-shot Learning

> **"좋은 AI는 좋은 프롬프트에서 시작된다."**

LLM은 동일한 모델이라도 프롬프트 설계 방식에 따라 결과의 품질이 크게 달라진다.
본 프로젝트에서는 회의록 요약 업무의 정확성과 일관성을 높이기 위해 **Zero-shot**, **Few-shot**, **Reasoning Prompt**를 단계적으로 적용하였다.

---

# 🎯 Prompt Engineering 적용 목적

이번 프로젝트에서 Prompt Engineering을 적용한 이유는 다음과 같다.

- 출력 형식을 일정하게 유지한다.
- 환각(Hallucination)을 최소화한다.
- 필요한 정보가 부족하면 먼저 질문하도록 유도한다.
- 회의록의 핵심 정보만 추출하도록 한다.
- 동일한 업무를 반복 수행해도 일관된 결과를 생성하도록 한다.

---

# 📌 Zero-shot Prompt

Zero-shot은 예시를 제공하지 않고 바로 작업을 수행하도록 요청하는 방식이다.

### Prompt

```text
아래 회의 내용을 요약해 주세요.

출력 형식

1. 결정사항
2. Action Items
3. 리스크
4. 후속 일정
```

### 결과

- 기본적인 요약은 가능하였다.
- Action Items가 누락되는 경우가 있었다.
- 출력 형식이 매번 조금씩 달랐다.
- 리스크를 추측하는 경우가 발생하였다.

---

# 📌 Few-shot Prompt

Few-shot은 AI에게 **좋은 입력과 좋은 출력 예시**를 먼저 보여준 뒤 동일한 방식으로 작업을 수행하도록 하는 방법이다.

이번 프로젝트에서는 출력 형식의 일관성을 확보하기 위해 Few-shot을 적용하였다.

---

# ✨ Few-shot Example 1

## ✅ Input

```text
[회의 제목]

UI 디자인 회의

[회의 내용]

- 로그인 화면 디자인은 4월 30일까지 완료한다.
- 개발은 디자인 완료 후 시작한다.
- QA 일정은 추후 결정한다.
```

---

## ✅ Output

```text
📌 결정사항

- 로그인 화면 디자인을 4월 30일까지 완료한다.
- 개발은 디자인 완료 후 시작한다.

✅ Action Items

|담당|업무|기한|
|디자인팀|로그인 화면 제작|4월 30일|

⚠️ 리스크

- QA 일정이 아직 결정되지 않았다.

📅 후속 일정

- QA 일정은 다음 회의에서 확정한다.

❓ 확인 필요 사항

- QA 담당자
```

---

## 💡 적용 효과

- 형식이 일정하게 유지됨
- 리스크가 명확하게 구분됨
- 확인 필요 사항을 별도로 출력

---

```
↓

3단계

↓

4단계

리스크 확인

↓

5단계

↓

6단계

```

```text
- Action Items
- 리스크
- 후속 일정
- 확인 필요 사항

만 출력합니다.

내부적인 추론 과정은 출력하지 않습니다.
```

# 🔍 Chapter 5. Hallucination(환각) 검증

> **"좋은 AI는 모르는 것을 아는 척하지 않는다."**

업무 자동화에서 가장 중요한 요소 중 하나는 **Hallucination(환각)** 을 최소화하는 것이다.

회의록, 정책 문서, 계약서, 보고서와 같이 **사실(Fact)** 을 기반으로 하는 업무에서는 AI가 근거 없이 정보를 생성할 경우 심각한 업무 오류로 이어질 수 있다.

따라서 본 프로젝트에서는 Hallucination의 기준을 명확하게 정의하고, 실제 검증 질문을 통해 AI의 신뢰성을 평가하였다.

---

# 🎯 Hallucination이란?

본 프로젝트에서는 Hallucination을 다음과 같이 정의하였다.

> **"검증 가능한 사실(날짜, 정책, 수치, 계산, 인물, 일정 등)에 대해 근거 없이 틀린 정보를 사실처럼 단정하여 생성하는 것."**

즉,

AI가

- 없는 회의 내용을 만들거나
- 존재하지 않는 담당자를 지정하거나
- 날짜를 임의로 생성하거나
- 정책을 추측해서 답하는 경우

모두 Hallucination으로 판단한다.

---

# ✍️ 창의적 생성과 Hallucination의 차이

Hallucination과 창의적 생성은 서로 다른 개념이다.

|구분|Hallucination|창의적 생성|
|:---|:---|:---|
|목적|사실 전달|창작 및 아이디어 생성|
|정답 존재 여부|존재|존재하지 않음|
|근거 필요|필수|불필요|
|허용 여부|❌ 허용되지 않음|✅ 허용 가능|
|예시|없는 정책을 생성|소설, 광고 문구 작성|

예를 들어,

### ❌ Hallucination

> "회의에서 QA는 5월 20일부터 시작하기로 결정되었습니다."

하지만 실제 회의록에 QA 일정이 없다면 이는 환각이다.

---

### ✅ 창의적 생성

> "미래의 AI 회사를 배경으로 한 소설을 작성해 주세요."

이 경우는 정답이 존재하지 않는 창작 작업이므로 Hallucination으로 보지 않는다.

즉, **사실 기반 업무와 창작 업무는 평가 기준이 다르다.**

---

# 📋 Hallucination 검증 방법

본 프로젝트에서는 아래 기준으로 검증을 수행하였다.

1. 동일한 질문을 입력한다.
2. AI의 답변을 확인한다.
3. 원문과 비교한다.
4. Pass / Fail을 판정한다.

---

# ✅ Pass 판정 기준

다음 두 조건 중 하나를 만족하면 **Pass**로 판정하였다.

### (A)

정답을 정확하게 제시하고

간단한 근거를 함께 설명한다.

예시

> "QA 일정은 회의록에 포함되어 있지 않습니다."

---

### (B)

정보가 부족한 경우

모른다고 답하고

확인 절차를 제안한다.

예시

> "회의록에 해당 내용이 없습니다.
원본 회의록이나 추가 자료를 확인해 주세요."

---

# ❌ Fail 판정 기준

다음 중 하나라도 해당하면 **Fail**로 판정하였다.

### ①

없는 사실을 생성한다.

예시

> "QA 일정은 5월 15일입니다."

(회의록에는 존재하지 않음)

---

### ②

근거 없이 확신한다.

예시

> "예산은 반드시 2천만원입니다."

(근거 없음)

---

### ③

질문이 모호한데도 확인 질문 없이 답한다.

예시

사용자

> "회의 일정 알려줘."

AI

> "다음 주 월요일입니다."

(어떤 회의인지 알 수 없음)

---

### ④

회의록에 없는 담당자를 생성한다.

예시

> "김철수 과장이 QA를 담당합니다."

(원문에 존재하지 않음)

# 📈 Hallucination 감소 전략

Prompt Engineering을 통해 다음과 같은 전략을 적용하였다.

### 1️⃣ 정보 부족 시 확인 질문

충분한 정보가 없으면 최대 3개의 질문을 먼저 수행한다.

---

### 2️⃣ 모르면 모른다고 답하기

추측 대신

> "확인이 필요합니다."

를 출력한다.

---

### 3️⃣ 원문 우선 원칙

회의록에 없는 내용은 절대 생성하지 않는다.

---

### 4️⃣ 출력 형식 고정

결정사항과 확인 필요 사항을 구분하여 작성한다.

---

### 5️⃣ 근거 중심 답변

수치, 일정, 정책은 반드시 원문에 근거하여 작성한다.

---

Hallucination을 완전히 제거하는 것은 어렵지만,

**추측을 줄이고 확인 질문을 유도하는 프롬프트 설계**를 통해 발생 가능성을 크게 낮출 수 있다.

업무 자동화에서는 AI의 답변을 그대로 사용하는 것이 아니라,

최종적으로 사람이 원문과 비교하여 검토해야 한다.

특히

- 계약
- 법률
- 정책
- 예산
- 일정

과 같은 정보는 반드시 원문 문서를 확인해야 한다.

# 🧠 문맥 유지 검증

LLM은 변경된 조건만 수정하고, 기존 요구사항은 그대로 유지하는지 확인하였다.

# ⚠️ 문제 발생 사례

### 문제 1

초기 Prompt(v1)에서는

담당자가 없는 경우

AI가 임의의 담당자를 생성하려는 경향이 있었다.

---

### 개선

Prompt에

> "회의록에 없는 담당자는 생성하지 않는다."

를 추가하였다.

---

### 문제 2

사용자가

"조금 더 간결하게"

라고 요청했을 때

기존 형식을 변경하는 문제가 있었다.

---

### 개선

Prompt에

> "형식은 유지하고 문장만 간결하게 수정한다."

라는 규칙을 추가하였다.

---

### 문제 3

조건이 여러 번 변경되면서

일부 모델은 이전 요구사항을 잊는 문제가 발생하였다.

---

### 개선

Prompt를 다음과 같이 변경하였다.

```text
새로운 요청이 들어오면

변경된 조건만 수정하고

기존 요구사항은 그대로 유지한다.
```

이를 통해 긴 대화에서도 일관성을 유지할 수 있었다.

---

# 🚀 문맥 유지를 위한 프롬프트 구조 개선

10턴 이상의 대화에서는 단순한 질의응답보다 **상태(State)를 유지하는 프롬프트 구조**가 중요하다.

본 프로젝트에서는 다음 원칙을 적용하였다.

1. **확정된 요구사항 유지**
   - 이미 합의된 형식, 톤, 금지사항은 이후 대화에서도 계속 유지한다.

2. **변경된 조건만 수정**
   - 새로운 요청이 들어오면 해당 부분만 업데이트하고 나머지 내용은 유지한다.

3. **최신 정보 우선 적용**
   - 사용자가 일정이나 예산을 수정하면 이전 정보를 덮어쓰고 최신 정보를 기준으로 답변한다.

4. **최종 출력 전 전체 검토**
   - 마지막 답변에서는 이전 대화의 모든 변경사항이 반영되었는지 확인한 후 결과를 생성한다.

이와 같은 구조는 긴 대화에서 문맥이 끊기는 문제를 줄이고, 업무 자동화에서 안정적인 결과를 유지하는 데 효과적이었다.

# 🚀 Chapter 7. 운영 전략 및 프로젝트 개선 방안

> **"좋은 프롬프트는 한 번 만드는 것이 아니라 지속적으로 개선하고 운영하는 것이다."**

Prompt Engineering은 단순히 프롬프트를 작성하는 것으로 끝나지 않는다.

실제 업무에서는

- 비용
- 모델 변경
- 업무 정책 변경
- 긴 문맥 유지
- 환각(Hallucination)

등 다양한 요소를 지속적으로 관리해야 한다.

본 프로젝트에서는 이러한 실제 운영 환경을 고려하여 업무 자동화 전략을 설계하였다.

---

# 💰 1. 비용 제약 환경에서의 운영 전략

실무에서는 항상 최신 유료 모델만 사용할 수 있는 것은 아니다.

기업이나 개인의 예산에 따라 무료 모델을 사용해야 하는 상황도 자주 발생한다.

따라서 무료 모델에서도 일정 수준 이상의 결과를 얻을 수 있도록 프롬프트를 개선하는 전략이 필요하다.

---

## 🎯 전략 1. 출력 형식을 명확하게 지정하기

무료 모델은 출력 형식이 자주 변경되는 경우가 있다.

이를 방지하기 위해 출력 템플릿을 프롬프트에 명확하게 포함하였다.

### Before

```text
회의 내용을 요약해 주세요.
```

---

### After

```text
회의 내용을 아래 형식으로 작성하세요.

1. 결정사항

2. Action Items

3. 리스크

4. 후속 일정

5. 확인 필요 사항

반드시 위 순서를 유지합니다.
```

---

## 🎯 전략 2. 긴 문서를 여러 단계로 나누기

무료 모델은 긴 입력에서 성능이 저하될 수 있다.

따라서

```
회의록

↓

핵심 내용 추출

↓

결정사항 생성

↓

Action Item 생성

↓

최종 요약
```

처럼 단계적으로 처리하도록 설계하였다.

---

## 🎯 전략 3. Few-shot 적극 활용

무료 모델은 출력 형식이 일정하지 않은 경우가 많다.

이를 해결하기 위해

- 좋은 입력
- 좋은 출력

예시를 함께 제공하여 출력 형식을 학습하도록 하였다.

---

## 🎯 전략 4. 불필요한 요청 제거

무료 모델에서는 한 번에 너무 많은 작업을 요청하면 품질이 떨어질 수 있다.

따라서

❌ 요약 + 번역 + 표 생성 + 일정 관리

처럼 여러 작업을 동시에 수행하지 않고,

각 작업을 분리하여 요청하도록 설계하였다.

---

## 📊 비용 절감 전략 요약

|전략|효과|
|:---|:---|
|출력 형식 고정|형식 오류 감소|
|단계적 처리|정확도 향상|
|Few-shot 적용|일관성 향상|
|작업 분리|응답 품질 향상|
|필요한 작업만 요청|토큰 비용 절감|

# 🛡️ 2. 정책 변경 환경에서의 Hallucination 감소 전략

기업의 업무 규정은 지속적으로 변경된다.

예를 들어

- 출장비 규정 변경
- 보안 정책 변경
- 결재 절차 변경
- 프로젝트 일정 변경

등은 자주 발생한다.

만약 AI가 오래된 정보를 그대로 답변한다면 업무 오류가 발생할 수 있다.

따라서 정책 변경을 고려한 운영 전략을 설계하였다.

---

## 🎯 운영 원칙

### ① 최신 문서를 우선 사용

AI는 기억에 의존하지 않고

최신 회의록,

최신 정책,

최신 업무 문서를 우선 참고하도록 한다.

---

### ② 정책 버전 확인

답변 전

```text
현재 사용하는 정책 버전을 확인합니다.
```

라는 절차를 추가하였다.

---

### ③ 변경 여부 확인 질문

정책이 자주 변경되는 업무에서는

AI가 먼저 질문하도록 설계하였다.

예시

```text
현재 사용하는 정책이

2025년 버전인가요?

2026년 개정판인가요?
```

---

### ④ 근거 없는 답변 금지

정책 문서가 없는 경우

```text
확인이 필요합니다.
```

라고 답하도록 하였다.

---

### ⑤ 최신 문서 기반 재생성

정책이 변경되면

이전 답변을 수정하는 것이 아니라

최신 문서를 기준으로 다시 생성하도록 설계하였다.

---

## 📊 정책 변경 대응 프로세스

```
사용자 질문

↓

최신 정책 존재?

↓

YES

↓

정책 기반 답변

↓

NO

↓

확인 질문

↓

최신 문서 요청

↓

답변 생성
```

# 🔄 Prompt 운영 개선 과정

Prompt는 한 번 작성한 이후에도 지속적으로 개선하였다.

---

## Version 1

```text
회의록을 요약해 주세요.
```

### 문제점

- 출력 형식이 일정하지 않음
- 긴 문서에서 누락 발생
- 정책 변경 대응 불가능

---

## Version 2

```text
회의록을 분석합니다.

먼저

결정사항을 추출합니다.

정보가 부족하면 질문합니다.

회의록에 없는 내용은 생성하지 않습니다.

출력 형식을 반드시 유지합니다.
```

---

## Version 3

```text
최신 회의록을 기준으로 분석합니다.

정책이나 일정이 변경되었는지 확인합니다.

정보가 부족하면 최대 3개의 질문을 수행합니다.

회의록에 없는 정보는 생성하지 않습니다.

최종 출력에서는

결정사항

Action Items

리스크

후속 일정

확인 필요 사항

만 출력합니다.
```
